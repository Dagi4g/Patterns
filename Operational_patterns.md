Here are the patterns for the operational design—the cashier and kitchen layers that scare everyone else away.

The first is the Event-Driven State Machine. An order isn't a database row that gets edited. It's a thing that moves through states: pending, confirmed, preparing, ready, completed, cancelled. Each state transition is an event. The cashier confirms payment. That's one event. The kitchen starts preparing. That's another. The kitchen marks it ready. Another. You don't update the order by setting fields. You define the valid transitions and trigger events. The order object listens for events and moves itself. This eliminates spaghetti code where any view can change the status to anything. The cashier can't accidentally mark an order as "ready." The kitchen can't un-confirm a payment. The state machine enforces the rules, and the code becomes a clean map of the real-world workflow.

The second is the Idempotent WebSocket Push. The kitchen display relies on WebSockets. Networks drop. Servers restart. The kitchen display must never miss an order. The pattern is to send every status update with a sequence number or timestamp. When the kitchen reconnects, it sends the last sequence it received. The server replays everything that happened since that moment. This ensures eventual consistency. The display might be briefly stale, but it will converge on the truth without manual intervention.

The third is the Outbox Pattern for WebSocket Delivery. When the cashier confirms an order, you write to the database and send a WebSocket message. What happens if the WebSocket fails after the database write but before the message is sent? The order is confirmed but the kitchen never knows. The fix is to write the notification to an outbox table in the same database transaction as the order update. A separate process, maybe a Django management command triggered by a database signal, reads the outbox and sends the WebSocket. If the send fails, it retries. Guaranteed delivery.

The fourth is the Optimistic Concurrency for Chair Booking. Two customers scan the same QR code within seconds. Both see the chair as available. Both try to order. The pattern is to add a version column to the Chair model. When you read it, you get version 3. When you update it, you say "update where version=3 and set version=4." If someone else updated it first, the version changed, your update fails, and you return an error. One of them gets the chair. The other sees "Sorry, this seat was just taken." No double bookings.

The fifth is the Command Query Responsibility Segregation for the Kitchen Display. The kitchen needs to see orders grouped by status—a column for "new," a column for "preparing," a column for "ready." Querying this from the Order table every second is wasteful. The pattern is to build a materialized view specifically for the kitchen that is updated by events. When an order changes status, the event updates the view. The display reads this pre-built structure instantly. The dashboard can afford a slow query; the kitchen cannot.

These patterns don't make the system less scary. They make it reliable. The ghost doesn't avoid complexity; he applies proven patterns that turn chaotic real-world workflows into predictable, fault-tolerant systems.
